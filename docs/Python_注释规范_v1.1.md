# Python 注释规范（量化团队版 · v1.1）

**适用范围：**
量化研究、策略研发、交易系统、工作流调度、数据工程等 Python 代码。

## 1. 注释设计总则（写给 AI 与人）

本规范旨在统一 Python 注释风格，让：

- AI 能准确理解文件结构、职责、上下文、数据关系
- 开发者能快速理解文件职责与逻辑
- 团队长期保持一致性、可维护性

原则：

- 注释必须准确、简洁、有信息增量。
- 所有注释均为中文主导，必要时使用英文术语（factor、signal、context）。
- 所有注释块采用一致结构，不同文件、不同开发者产出的注释风格应保持统一。
- 不重复代码内容，重点说明“为什么这样写”。
- 注释结构应统一，让 AI 能可靠读取与复用。
- 文件、类、函数、模块级注释均采用一致的风格模式。

### 1.1 语言与标点规范（建议写入团队规则）

- 注释内容以中文为主；英文仅用于必要的技术/业务术语、缩写、协议名、字段名等。
- 专有名词、代码标识符（类名、函数名、变量名、列名）保持原样，不强行翻译。
- 中文与英文/数字混排时，建议两侧各空一格（例如：“使用 DatetimeIndex 对齐交易日”）。
- 中文标点使用全角（例如：，。）；英文缩写内部按英文规则（例如：e.g.、i.e.）。
- 避免口语化、情绪化表达；用事实描述约束与意图。

## 2. 文件头注释规范（PEP 风格 + AI 友好依赖契约）

文件头注释（Module Docstring）是整个文件最重要的注释，供 AI 和开发者理解模块定位。

### 2.1 格式要求（必须）

```python
"""
一句话 Summary。

详细说明……
"""
```

要求：

- 文件头注释必须为文件中的第一个语句，位于所有 import 之前。
- 使用三重双引号，首尾单独一行。
- Summary 必须为文件的第一行。
- Summary 后空一行。
- 注释不得包含 emoji、特殊符号（★、❤、→ 等）。
- 每行不超过 100 字符（必要时换行）。
- 列表统一使用 `- `（短横线后空一格）。

### 2.2 文件头注释结构（推荐）

建议包含以下小节（按需省略非必需项）：

- Summary（必须）
- Responsibilities（必须）：模块承担的职责（列表）
- Data Contract（条件必选 + 强制兜底说明）
- Internal Dependencies（必须）
- External Systems（必须）
- Upstream Usage（推荐）
- Notes（可选）

### 2.3 Data Contract（数据契约）规则

Data Contract 采用“条件必选 + 强制兜底说明”：

- 当满足以下条件时必须写：
  - 文件的主要函数对输入数据结构有统一前提
  - 文件属于信号计算、因子计算、风险控制等结构化模块
- 当文件内部存在多种数据格式、不具备统一结构时，文件头必须写兜底说明：

```text
Data Contract:
    - 本模块包含多类逻辑，不对输入数据结构做统一约束
    - 具体数据结构与约束见各函数 docstring
```

并要求：

- 函数 docstring 必须清楚描述各自的数据要求。
- 工作流文件、聚合脚本优先写“全局约定”（如时区、金额单位、频率范围），细节落在分区或函数级 docstring。

### 2.4 文件头注释完整示例

```python
"""
ETF 轮动策略的动量与趋势信号模块。

Responsibilities:
    - 提供 N 日动量、波动率等因子计算函数
    - 基于动量与波动率生成标准化交易信号
    - 为策略核心模块提供指标计算接口

Data Contract:
    - df 参数必须包含列：'open', 'high', 'low', 'close', 'volume'
    - df.index 必须为升序 DatetimeIndex，不得重复
    - 输入数据为日频，不支持分钟级数据
    - 若存在缺失值，调用方必须在上游完成清洗与填补

Internal Dependencies:
    - utils.signal: 提供信号平滑与标准化工具
    - infra.logger: 提供统一日志记录接口

External Systems:
    - None

Upstream Usage:
    - strategy_core: 获取每日交易信号
    - backtest_runner: 回测因子计算使用

Notes:
    - 本模块在回测中高频调用，应避免循环内大量对象创建
"""
```

## 3. 类注释规范（Google 风格 · 团队版）

类注释说明类的职责、用途和关键属性。

### 3.1 格式要求

- 紧跟在 class 定义下方。
- `"""` 单独占行。
- 第一行为 Summary。
- Summary 与后续段落之间必须空一行。

### 3.2 类注释结构（推荐）

- Summary（必须）
- Note / 额外说明（可选）
- Attributes（建议在关键类中写）
- Examples（可选，仅例外情况）

### 3.3 Examples 规范

示例（Examples）非必需。仅当类为对外公共接口、复杂用法较多时，建议增加示例，由模块负责人决定。
默认内部类不写 Examples，以避免维护负担。

### 3.4 类注释示例

```python
class RiskManager:
    """
    全局风控管理器，负责控制最大回撤与仓位上限。

    用于回测与实盘中统一管理组合风险，对接策略信号与订单执行模块。

    Attributes:
        max_drawdown (float): 最大回撤限制。
        max_position (float): 最大仓位比例。
        symbol_limit (float): 单一标的最大仓位比例。
    """
```

## 4. 函数/方法注释规范（Google 风格 · 团队版）

函数注释用于说明函数的目的、参数、返回结构。

### 4.1 格式要求

- 紧跟 def 定义后。
- 三重双引号，首尾另占一行。
- 第一行为 Summary。
- 必须包含 Args，建议包含 Returns。
- Note 可选。
- 不要求 Raises（如需说明异常行为，用 Note 自然表达）。
- 当同时包含 Args: 与 Returns: 时，在 Args 小节结束后空一行，再开始 Returns:

### 4.2 Returns 规范

- 必须说明返回类型和含义，特别是：
  - pandas Series / DataFrame
  - 数值型指标
  - 字典结构

### 4.3 函数注释示例

基础示例：

```python
def compute_momentum(df: pd.DataFrame, period: int = 20) -> pd.Series:
    """
    计算指定周期的价格动量。

    Args:
        df (pd.DataFrame): 输入数据，必须包含 'close' 列。
        period (int): 动量计算周期。

    Returns:
        pd.Series: 与 df.index 对齐的动量序列。
    """
```

带 Note 的示例：

```python
def generate_signals(momentum: pd.Series, vol: pd.Series, mom_th: float, vol_th: float) -> pd.Series:
    """
    根据动量与波动率阈值生成多空信号。

    该函数仅生成基础信号，不包括持仓状态平滑逻辑。

    Args:
        momentum (pd.Series): 动量指标。
        vol (pd.Series): 波动率指标。
        mom_th (float): 动量阈值。
        vol_th (float): 波动率阈值。

    Returns:
        pd.Series: 多空信号（1 多头，-1 空头，0 空仓）。
    """
```

## 5. 模块级分区注释规范（文件内部章节划分）

用于将同一文件内的不同逻辑区域分组。

### 5.1 分区格式

统一采用：

```text
# ---- 中文标题 ----
```

规则：

- 横线固定为 4 个 `-`。
- 标题使用中文为主，必要时添加英文补充。
- 分区注释上下方各空一行。
- 在类或函数内部也可使用，遵守缩进。

示例：

```python
# ---- 数据加载 ----

def load_raw_data(...):
    ...
```

## 6. 代码块注释规范（Block Comments）

代码块注释解释一段连续代码的目的。

### 6.1 普通逻辑（不编号）

```python
# 根据过去 N 日收益率计算滚动最大回撤
rolling_max = ...
```

要求：

- 用一句或短段描述逻辑意图。
- 不重复代码所表达的内容。

### 6.2 工作流 / 复杂流程（编号规则）

适用于数据流程、任务编排、批处理脚本等。

格式：

```python
# 1. 加载原始数据
# 2. 进行清洗与对齐
# 3. 计算各类因子
# 4. 保存结果
```

规则：

- 使用 `1.` `2.` `3.` 这种简单英文编号。
- 每一步必须是一个明确的动作。
- 用中文说明目的，保持一致风格。

## 7. 行内注释规范（Inline Comments）

用于解释某一行的特殊逻辑。

### 7.1 行内注释规则

- `#` 后必须空一格。
- 长行不超过约 100 字符。
- 注释应说明“为什么这样写”，不要复述代码含义。

示例：

```python
risk_cap = 0.02  # 单笔最大允许亏损 2%
```

## 8. TODO / FIXME 注释规范

用于标注未来工作或现存问题。

### 8.1 写法

```python
# TODO: 支持分钟级数据
# FIXME: 此处阈值需要根据费用重新评估
```

规则：

- 必须使用大写 TODO / FIXME。
- 描述必须明确、可执行。
- 处理完毕后必须删除。

---

## 9. 禁止事项（全部注释类型适用）

- 禁止 emoji、表情、特殊符号（★、❤）。
- 禁止使用三重引号作为块注释：“未赋值字符串注释”。
- 禁止写“无信息注释”：

```python
# 计算收益   # 无意义
```

- 注释不得与代码行为不符。
- 注释不得用于解释明显语义的代码。
